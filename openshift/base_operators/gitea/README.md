# Install Red Hat Gitea Operator on OpenShift

This folder contains an Ansible playbook that installs the Red Hat Gitea Operator and deploys a Gitea instance into a dedicated namespace.

## Gitea Note

Use Gitea to create an internal Git source for onboarding repositories only when an existing Git service is not available.

It is not recommended to use public Git repositories for this workflow, because much of the repository content is specific to each customer environment and can include sensitive infrastructure configuration details.

## What the Playbook Does

The playbook in this folder performs the following workflow:

1. Logs in to OpenShift using environment-provided token and API URL.
2. Creates namespace gitea-operator.
3. Applies OperatorGroup in that namespace.
4. Applies a namespace-scoped CatalogSource named redhat-rhpds-gitea.
5. Applies a Subscription named gitea-operator (channel: stable).
6. Waits for deployment gitea-operator-controller-manager to become available.
7. Applies a Gitea custom resource named gitea-with-admin.

## Prerequisites

- OpenShift CLI (oc) installed on the host running Ansible
- Ansible with collection kubernetes.core installed
- Cluster credentials with permission to create namespace, OLM resources, and CRs
- Network access from cluster nodes to pull the catalog image:
  - quay.io/rhpds/gitea-catalog:latest

Install required collection:

```bash
ansible-galaxy collection install kubernetes.core
```

## Required Environment Variables

The playbook reads OpenShift access values from environment variables:

```bash
export openshift_api_url="https://api.<cluster>.<domain>:6443"
export openshift_token_id="<token>"
```

Get token and API URL from the OpenShift web console:

- Click user menu (top-right)
- Select Copy login command
- Select Display Token

## Validation

Check namespace and operator components:

```bash
oc get ns gitea-operator
oc get operatorgroup -n gitea-operator
oc get catalogsource -n gitea-operator
oc get subscription -n gitea-operator
oc get csv -n gitea-operator
oc get deployments -n gitea-operator
```

Check Gitea custom resource and resulting pods/services:

```bash
oc get gitea -n gitea-operator

oc get svc -n gitea-operator
```

## Template Defaults

The default Gitea instance template currently sets:

- giteaSsl: true
- giteaAdminUser: opentlc-mgr
- giteaAdminPassword: "" (auto-generated by operator)
- giteaAdminPasswordLength: 32
- giteaAdminEmail: opentlc-mgr@redhat.com
- giteaCreateUsers: true
- giteaGenerateUserFormat: lab-user
- giteaUserNumber: 1
- giteaUserPasswordLength: 16

If needed, adjust templates before execution.

## Troubleshooting

- oc login failure:
  - Confirm openshift_api_url and openshift_token_id are exported in the same shell.
- Operator deployment never becomes ready:
  - Check subscription and CSV status in gitea-operator namespace.
  - Confirm cluster can pull quay.io/rhpds/gitea-catalog:latest.
- Gitea CR exists but pods are missing:
  - Inspect operator logs from gitea-operator-controller-manager deployment.
  - Describe the Gitea CR for status and events.

Helpful commands:

```bash
oc describe subscription gitea-operator -n gitea-operator
oc get csv -n gitea-operator
oc logs deployment/gitea-operator-controller-manager -n gitea-operator
oc describe gitea gitea-with-admin -n gitea-operator
oc get events -n gitea-operator --sort-by=.metadata.creationTimestamp
```
